# 错误码设计文档

## 设计原则

### 1. 用户友好优先
- **用户可见错误 (1000-1999)**: 返回清晰、友好的错误消息
- **内部错误 (5000-5999)**: 统一返回"系统繁忙，请稍后重试"
- **业务错误 (6000-6999)**: 根据类型返回通用消息

### 2. 安全性
- 不向用户暴露系统内部细节
- 数据库错误、缓存错误等统一处理
- 详细错误信息仅记录在服务器日志

### 3. 可维护性
- 错误码分类清晰
- 内部错误码帮助快速定位问题
- 日志记录完整的错误上下文

## 错误码分类

### 用户可见错误 (1000-1999)

这些错误会直接返回给前端用户，需要提供清晰的错误消息。

| 错误码 | 用户看到的消息 | 使用场景 |
|--------|---------------|----------|
| 1001 | 用户名或密码错误 | 登录失败 |
| 1002 | 请先登录 | Token 无效或过期 |
| 1003 | 您没有权限访问此资源 | 权限不足 |
| 1004 | 您的账号已被禁用 | 账号被封禁 |
| 1101 | 输入内容不符合要求 | 参数验证失败 |
| 1102 | 请填写所有必填项 | 缺少必填字段 |
| 1103 | 该资源已存在 | 用户名/邮箱重复 |
| 1201 | 请求的资源不存在 | 通用资源不存在 |
| 1202 | 页面不存在或已被删除 | 页面不存在 |
| 1301 | 操作失败，请稍后重试 | 通用操作失败 |
| 1302 | 操作过于频繁 | 限流 |
| 1303 | 已超出配额限制 | 超出限制 |
| 1304 | 不允许执行此操作 | 业务规则限制 |

### 内部错误码 (5000-5999)

这些错误不会暴露给用户，统一返回"系统繁忙，请稍后重试"。

**数据库错误 (5000-5099)**
- 5001: 数据库连接失败
- 5002: 数据库查询错误
- 5003: 数据库插入失败
- 5004: 数据库更新失败
- 5005: 数据库删除失败
- 5006: 数据库事务失败

**缓存错误 (5100-5199)**
- 5101: 缓存连接失败
- 5102: 缓存读取失败
- 5103: 缓存写入失败
- 5104: 缓存删除失败

**外部服务错误 (5200-5299)**
- 5201: 外部服务调用失败
- 5202: 邮件服务失败
- 5203: 存储服务失败

**系统错误 (5900-5999)**
- 5901: 服务器内部错误
- 5902: 配置错误
- 5999: 未知错误

### 业务逻辑错误 (6000-6999)

用于内部定位具体业务问题，返回给用户时转换为通用消息。

**用户业务 (6000-6099)** → 用户看到: "登录已过期" 或 "用户信息错误"
- 6001: 用户不存在
- 6002: 用户已存在
- 6003: 邮箱已存在
- 6004: Token 无效
- 6005: Token 过期

**工作空间业务 (6100-6199)** → 用户看到: "无权访问该工作空间"
- 6101: 工作空间不存在
- 6102: 无权访问工作空间
- 6103: 不是工作空间所有者

**页面业务 (6200-6299)** → 用户看到: "页面不存在或已被删除"
- 6201: 页面不存在
- 6202: Slug 已存在
- 6203: Slug 格式错误

**Block 业务 (6300-6399)** → 用户看到: "内容不存在或已被删除"
- 6301: Block 不存在
- 6302: Block 类型无效
- 6303: 同步失败

## 使用示例

### Service 层返回错误

```go
// 数据库查询失败 - 内部错误
user, err := s.userRepo.FindByID(id)
if err != nil {
    if err == gorm.ErrRecordNotFound {
        // 业务错误：用户不存在
        return nil, errors.New(errors.ErrUserNotFound, "user not found in database")
    }
    // 数据库错误
    return nil, errors.Wrap(errors.ErrDatabaseQuery, err)
}

// 用户可见错误
if user.Password != hashedPassword {
    return nil, errors.New(errors.ErrInvalidCredentials, "password mismatch")
}
```

### Handler 层处理错误

```go
user, err := h.userService.Login(username, password)
if err != nil {
    // 统一错误处理，自动区分用户可见和内部错误
    response.HandleError(c, err)
    return
}
response.Success(c, user)
```

### 实际返回示例

**场景 1: 用户登录密码错误**
```json
{
  "code": 1001,
  "message": "用户名或密码错误"
}
```
服务器日志: 无（正常业务逻辑）

**场景 2: 数据库连接失败**
```json
{
  "code": 5001,
  "message": "系统繁忙，请稍后重试"
}
```
服务器日志: `[ERROR] Code: 5001, Error: dial tcp: connection refused`

**场景 3: 用户不存在（业务错误）**
```json
{
  "code": 6001,
  "message": "登录已过期，请重新登录"
}
```
服务器日志: `[ERROR] Code: 6001, Detail: user not found in database`

## 前端处理建议

```typescript
try {
  const response = await api.post('/auth/login', { username, password })
  // 成功处理
} catch (error) {
  const { code, message } = error.response.data
  
  // 直接显示服务器返回的消息（已经是用户友好的）
  showError(message)
  
  // 特殊处理某些错误码
  if (code === 1002) {
    // 未登录，跳转登录页
    router.push('/login')
  }
}
```

## 监控和告警

### 需要告警的错误
- 5000-5999: 所有内部错误都需要告警
- 频繁出现的 6000-6999: 可能表示业务逻辑问题

### 日志级别
- 1000-1999: INFO（用户操作错误，正常）
- 5000-5999: ERROR（系统错误，需要处理）
- 6000-6999: WARN（业务逻辑错误，需要关注）

## 扩展指南

### 添加新的用户可见错误
1. 在 `codes.go` 中定义错误码（1000-1999）
2. 在 `userFacingMessages` 中添加用户友好消息
3. 更新本文档

### 添加新的内部错误
1. 在 `codes.go` 中定义错误码（5000-5999）
2. 在 `internalMessages` 中添加内部消息
3. 无需更新用户文档（用户看不到）

### 添加新的业务错误
1. 在 `codes.go` 中定义错误码（6000-6999）
2. 在 `internalMessages` 中添加内部消息
3. 在 `getBusinessErrorUserMessage` 中配置用户消息映射
