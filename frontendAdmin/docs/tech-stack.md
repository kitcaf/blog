# 博客发布管理页面技术栈文档（类Notion）

## 1. 技术架构概览

### 1.1 核心技术决策

本项目采用 **Block 扁平化数据结构** + **Tiptap 编辑器** + **BlockNote 基础框架** 的技术方案，实现类似 Notion 的博客编辑体验。

### 1.2 技术栈层次结构

```
┌─────────────────────────────────────────────────────────┐
│                   表现层                                 │
│         React 18 + TypeScript + Tailwind CSS            │
└─────────────────────┬───────────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────────┐
│                   编辑器层                               │
│                                                          │
│  BlockNote (基础框架)                                    │
│  - 提供开箱即用的 Block 编辑器                           │
│  - 内置斜杠命令、拖拽、格式化等功能                       │
│  - 基于 Tiptap 构建，可完全定制                          │
└─────────────────────┬───────────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────────┐
│                   引擎层                                 │
│                                                          │
```

## 2. 为什么选择这个技术栈？

### 2.1 BlockNote 作为基础框架

**选择理由**:
- **开箱即用**: 提供完整的 Block 编辑器实现，大幅减少开发时间
- **基于 Tiptap**: 继承 Tiptap 的所有优势，同时提供更高层次的抽象
- **现代化**: 使用 React 18 和 TypeScript，类型安全
- **可定制性强**: 支持自定义 Block 类型、样式和行为
- **活跃维护**: 社区活跃，文档完善，持续更新
- **极简主义友好**: 可以完全定制 UI，去除默认样式

**核心优势**:
- 无需从零实现 Block 编辑逻辑
- 内置斜杠命令、拖拽排序、格式化工具栏
- 提供完整的 Block 数据结构
- 支持 Markdown 快捷输入
- 自带撤销/重做功能

### 2.2 Tiptap 作为核心引擎

**选择理由**:
- **React Node Views**: 每个 Block 可以是独立的 React 组件，便于定制
- **扩展性强**: 丰富的官方和社区扩展（图片、代码块、表格等）
- **性能优秀**: 基于 ProseMirror，处理大文档性能好
- **协同编辑**: 原生支持 Yjs 协同编辑（未来可扩展）
- **文档完善**: 官方文档详细，示例丰富
- **灵活性**: 可以完全控制编辑器行为

**核心优势**:
- 比 Lexical 更成熟，生态更丰富
- 比 Slate.js 更易用，学习曲线平缓
- 支持复杂的嵌套结构
- 提供强大的插件系统

### 2.3 Block 扁平化数据结构

**选择理由**:
- **简单直观**: 每个 Block 是一个独立的对象，易于理解
- **易于存储**: 直接存储为 PostgreSQL JSONB，无需复杂的序列化
- **易于查询**: 可以对 Block 内容进行 JSON 查询和全文搜索
- **易于渲染**: 前端直接遍历 Block 数组渲染，无需复杂的树形遍历
- **易于扩展**: 添加新 Block 类型只需扩展数据结构
- **易于迁移**: 数据格式清晰，便于导入导出

**数据结构特点**:
- 每个 Block 有唯一 ID
- 每个 Block 有类型（段落、标题、列表、图片等）
- 每个 Block 有属性（颜色、对齐、级别等）
- 每个 Block 有内联内容（文本、链接、格式化）
- 支持 Block 嵌套（列表嵌套、引用嵌套）

## 7. 性能优化策略

### 7.1 前端优化

- **代码分割**: 路由懒加载，组件按需加载
- **资源优化**: 图片懒加载，WebP 格式
- **缓存策略**: React Query 缓存，LocalStorage 草稿
- **虚拟滚动**: 长列表性能优化
- **防抖节流**: 搜索、自动保存

### 7.2 后端优化

- **数据库索引**: 常用查询字段索引
- **JSONB 索引**: GIN 索引加速查询
- **连接池**: 数据库连接复用
- **缓存**: Redis 缓存热点数据
- **并发处理**: Go 协程处理请求

### 7.3 编辑器优化

- **防抖保存**: 2-3 秒防抖，减少请求
- **本地草稿**: IndexedDB 存储，防止数据丢失
- **懒加载**: 图片和媒体懒加载
- **虚拟滚动**: 长文档性能优化

## 8. 安全性考虑

### 8.1 前端安全

- **XSS 防护**: React 自动转义，CSP 配置

- **敏感信息**: 不在前端存储敏感信息

### 8.2 后端安全

- **输入验证**: 所有 API 参数验证
- **SQL 注入**: ORM 参数化查询
- **认证授权**: JWT Token
- **限流**: API 限流防止暴力破解
- **文件上传**: 类型和大小限制

## 10. 总结

### 10.1 技术栈优势

1. **开发效率高**: BlockNote 提供开箱即用的编辑器
2. **可定制性强**: Tiptap 和 React 完全可控
3. **性能优秀**: Go 后端 + PostgreSQL + React
4. **易于维护**: TypeScript 类型安全 + 清晰的架构
5. **可扩展性好**: Block 数据结构易于扩展

### 10.2 技术选型原则

- **成熟稳定**: 选择经过验证的技术
- **社区活跃**: 便于解决问题和获取支持
- **文档完善**: 降低学习成本
- **性能优先**: 保证用户体验
- **可维护性**: 代码清晰，易于扩展
