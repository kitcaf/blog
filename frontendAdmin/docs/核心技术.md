
# 数据模型架构方案

开启 PostgreSQL 的 pgcrypto 或 uuid-ossp 扩展，并强烈建议在前端生成 UUID（这是实现本地优先、离线编辑的关键）

物化路径（Materialized Path） 结合 邻接表（parent_id） 和 子节点排序数组（content_ids） 的混合模式

```sql
CREATE TABLE blocks (
    id UUID PRIMARY KEY,                   -- 唯一标识 (前端生成)
    parent_id UUID,                        -- 父节点 ID (用于快速找直系父级)
    
    -- 核心：物化路径字段
    path VARCHAR(1000) NOT NULL,           -- 例如: '/root-uuid/parent-uuid/this-uuid/'
    
    type VARCHAR(50) NOT NULL,             -- 块类型: 'page', 'h1', 'paragraph', 'code_block' 等
    
    -- 核心：子节点排序 (数组形式)
    content_ids UUID[] DEFAULT '{}',       -- 记录其子块的 ID 数组，严格决定子块的展示顺序
    
    -- 核心：块的属性与内容 (高扩展性)
    properties JSONB DEFAULT '{}',         -- 存储文本内容、样式、图片URL、是否发布等所有动态属性
    
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    deleted_at TIMESTAMP WITH TIME ZONE    -- 软删除标识 (在同步架构中，切忌物理删除数据)
);

-- 建立核心索引
CREATE INDEX idx_blocks_path ON blocks USING btree (path varchar_pattern_ops);
CREATE INDEX idx_blocks_parent_id ON blocks(parent_id);
CREATE INDEX idx_blocks_type ON blocks(type);
```

### 痛点 1：如何一次性加载整个“侧边栏目录树”？

传统做法：递归查询 parent_id，查询 N 次数据库，极慢。
物化路径解法：只需要一条 SQL。
由于侧边栏只需要显示 page（不需要显示里面的段落 text），你的 Go 后端只需执行：
甚至后期可以api设置 GET /api/pages/tree?level=root
在加入一个level，对path或者parent_id进行筛选

```SQL
SELECT id, parent_id, properties 
FROM blocks 
WHERE type = 'page' 
  AND deleted_at IS NULL;
  ```
如果只要某一个特定文件夹（如 uuid-1）下的所有子孙页面：
```SQL
SELECT id, parent_id, properties 
FROM blocks 
WHERE type = 'page' 
  AND path LIKE '/uuid-1/%'; -- 命中索引，速度极快
 ```
注：前端拿到这批扁平的数据后，通过 parent_id 在内存里花 1 毫秒就能组装成一棵 Tree 用于渲染。

### 痛点 2：用户在前端拖拽，改变了段落的上下顺序，怎么办？
在 React (Tiptap) 中，用户把“段落 2(uuid-4)”拖到了“段落 1(uuid-3)”前面。
解法：根本不需要修改 uuid-3 或 uuid-4 的数据！
你只需要更新它们共同的父节点（uuid-2）的 content_ids 字段即可：

```SQL
UPDATE blocks 
SET content_ids = '{"uuid-4", "uuid-3"}' -- 顺序变了
WHERE id = 'uuid-2';
```
这就是为什么必须保留 content_ids 数组。path 解决的是“层级包含关系”，content_ids 解决的是“同一层级的严格排序”。

### 痛点 3：移动文件夹（把整个文件夹拖到另一个文件夹里）

假设你把 uuid-2(Go 并发编程) 及其里面的几百个字，整个拖到了一个全新的文件夹 uuid-99(后端开发) 下。
这也是物化路径稍微复杂的地方，但在工程上有一条标准的 SQL 可以瞬间完成批量替换：

```SQL
-- 将路径中旧的父节点前缀，替换为新的父节点前缀
UPDATE blocks
SET 
    parent_id = 'uuid-99', -- 更新直属父亲
    path = REPLACE(path, '/uuid-1/uuid-2/', '/uuid-99/uuid-2/') -- 批量更新自己和所有子孙的路径
WHERE path LIKE '/uuid-1/uuid-2/%';
```

# 编辑更新方案

## 方案一：**本地优先 (Local-First) + 防抖 (Debounce) + 批量同步 (Batch Sync)**

1. 前端状态即时响应（0 延迟）
当你在 Tiptap 编辑器里打字时，修改的是前端内存里的状态。你敲下键盘，屏幕立刻显示，这个过程完全不调用任何网络接口，所以体验如同本地原生软件一样丝滑。

2. 防抖同步 (Debounce)
前端会监听编辑器的内容变化（比如 Tiptap 的 onUpdate 事件），但会套上一层“防抖”逻辑。
规则是：“用户如果正在连续疯狂打字，先不要发请求。只有当用户停止打字超过 1.5 秒或 2 秒时，才触发一次保存接口。”

3. 批量更新 (Batch Updates)
防抖触发后，前端也不是把整篇文章发过去，而是收集这 2 秒内发生变动的 Block，打包成一个数组发给 Go 后端：
```
// POST /api/blocks/sync
{
  "updated_blocks": [
    { "id": "uuid-4", "content": "Gin 是一个非常快..." }, // 修改了的段落
    { "id": "uuid-5", "content": "它基于..." }            // 新增的段落
  ],
  "deleted_blocks": ["uuid-6"] // 删掉的段落
}
```
## 方案二：Yjs (CRDT) + WebSocket

> 协作相关问题

前端用 Yjs 绑定 Tiptap。
后台 Go 起一个 WebSocket 连接。

每次打字，Yjs 会在底层自动计算出极其微小的“二进制差异增量（Delta）”，默默通过 WebSocket 流式发送给服务器。这种二进制包极其轻量（可能只有几个字节），即使高频发送也完全不会对服务器造成压力。这就是多人实时协作能做到“光标乱飞也不卡”的秘密。

# 数据流架构

[ PostgreSQL / Go API ]

  数据形态：DbBlock (原始数据库结构)
        ↑ ↓

[ Zustand Store / Tiptap 内部状态 ]         <-- 工作台（负责高频打字、拖拽、毫秒级响应）
  数据形态：Block (前端强类型结构)
        ↑ ↓
[ React 编辑器 UI ]                       <-- 用户实际看到和操作的界面

# 核心编辑器原理

《单实例 + 节点追踪流派 (Single Instance + Node Tracking)》

在搞华丽的 UI 渲染器之前，我们必须先让 Tiptap 的底层拥有**“Block 意识”**。也就是我们之前聊的“注入灵魂”。

- 给基础节点加 ID：重写（Extend）Tiptap 自带的 Paragraph、Heading 等扩展，让它们在 attrs 里必须携带 id（UUID）。

- 拦截并生成 ID：写一个全局插件，监听 Tiptap 的内容分裂（比如按 Enter 键新建段落时），自动为新产生的空节点生成一个唯一的 UUID。

- 完成脱水与 Diff (dehydrate)：写出那个最关键的“压路机”函数。监听 onUpdate，提取带有 ID 的扁平数组，对比 Zustand 里的旧数据，计算出增量（Delta）准备发给后端。（增（Create）、删（Delete）、改（Update）、移（Move/Reorder））

## Diff算法：Zustand 状态机 + Tiptap onUpdate

- 我们并不依赖 onUpdate 去做全量 Diff。

- 既然我们写了自定义的 React 组件（NodeView）来渲染 Block，我们可以直接在 React 组件的 onChange 或 Tiptap 的细粒度事件里，精准捕获**“当前用户正在编辑哪个 ID”**。

- Zustand Store 维护一个 dirty_set = new Set<string>()。

- 用户在 block-5 里打字，Zustand 立刻把 block-5 塞进 dirty_set。

- 等待 1.5 秒（防抖触发）。

- Diff 引擎启动，它根本不看别的节点，直接从 Tiptap 里精准提取 block-5 的数据，发给后端！然后清空 dirty_set。

### 在 Zustand 里，我们其实维护两个集合：

dirty_set = new Set<string>() （记录谁被修改/新增了）

deleted_set = new Set<string>() （记录谁被彻底删除了）

1. 改 (Update) —— 你的例子，最常见的场景
动作：用户在 block-5 里疯狂打字。

Zustand：dirty_set.add('block-5')。

同步引擎：防抖结束后，直接从 Tiptap 提取 block-5 的最新 JSON，发给后端。

2. 增 (Create) —— 按下 Enter 键
动作：用户按下回车，Tiptap 插件瞬间生成了一个带新 UUID 的 block-6。

Zustand：

dirty_set.add('block-6') (新段落本身要同步)

【核心细节】 dirty_set.add('page-root')！(因为父页面的 contentIds 数组里多了一个兄弟，父页面也算被“修改”了！)

同步引擎：发送 block-6 的完整数据，以及 page-root 的新排序数组。

3. 移 (Move) —— 鼠标拖拽换行
动作：用户把 block-5 拖到了 block-2 前面。

Zustand：block-5 和 block-2 的内容其实都没变，真正改变的是父级！所以只需 dirty_set.add('page-root')。

同步引擎：向后端发送 page-root 更新后的 contentIds 即可，底层节点数据完全不需要走网络。

4. 删 (Delete) —— 按退格键删除了某一段
动作：用户按 Backspace，把 block-5 删没了。

Zustand：

deleted_set.add('block-5') (把它打入死牢)

dirty_set.add('page-root') (父页面的排序数组里少了它)

同步引擎：发送 page-root 的新排序数组，并在 API payload 中带上 deleted_blocks: ['block-5']。
