# Plan

本阶段目标是在 Profile & Projects 数据化阶段基本完成后，将站点从“本地可构建”推进到“可稳定自动发布”。当前项目已经具备构建期数据同步能力：Notion 文章、GitHub Projects、GitHub Profile 与贡献数据都可以在构建前生成静态 JSON，前端通过 SSG 输出 `frontend/dist`。

下一阶段不再继续扩展页面数据结构，而是围绕 `blogFlow.md` 中的两种部署方案建立完整发布链路：方式一是自建服务器定时构建与静态托管，方式二是 GitHub Actions 云端定时构建与静态托管。两种方案应共用同一套数据同步、类型检查和前端构建命令，避免出现两套部署逻辑。

## Requirements

- 部署阶段必须以 `frontend/dist` 作为唯一发布产物。
- 数据同步必须发生在构建期，不能让浏览器运行时请求 Notion 或 GitHub API。
- Token 只能存在于根 `.env`、服务器环境变量或 GitHub Secrets，不能使用 `VITE_` 前缀。
- 自建服务器方案需要支持定时构建、构建日志、失败不覆盖线上版本。
- GitHub Actions 方案需要支持手动触发和定时触发。
- 两种方案应复用同一套根命令，不能各自硬编码不同的数据同步流程。
- 构建成功后应可验证线上静态页面是否可访问。
- 构建失败时应保留上一版可用产物，避免线上空白。
- 发布方案需要为后续 SEO 文件、RSS、sitemap、robots 扩展预留位置。

## Scope

- In: 统一构建命令、服务器部署脚本设计、crontab/systemd timer 方案、Nginx/Caddy 静态托管方案、GitHub Actions 部署 job、GitHub Pages 或 Cloudflare Pages 发布策略、部署环境变量与 Secrets 设计、构建日志、回滚策略、基础健康检查。
- Out: SSR 服务端渲染、数据库、登录系统、在线 CMS、容器编排平台、复杂灰度发布、多区域部署、运行时 API 服务。

## Unified Build Pipeline

两种部署方案都应围绕同一条构建流水线：

```text
pnpm install --frozen-lockfile
pnpm sync:notion
pnpm sync:github
pnpm --filter frontend type-check
pnpm build:frontend
```

产物：

```text
frontend/dist
```

建议后续新增一个根命令来表达完整发布前构建：

```json
{
  "scripts": {
    "sync:data": "pnpm sync:notion && pnpm sync:github",
    "verify:frontend": "pnpm --filter frontend type-check",
    "build:site": "pnpm sync:data && pnpm verify:frontend && pnpm build:frontend"
  }
}
```

这样自建服务器和 GitHub Actions 都只需要调用同一个入口，后续增删同步步骤也只改一处。

## Deployment Option 1: Self-hosted Server

核心构建一个一个服务器部署脚本，通常是 bash 脚本

适用场景：

- 当前测试阶段。
- 希望 token 和构建过程完全掌握在自己服务器上。
- 后续可能需要接入自定义图床、私有网络、服务端脚本或更多本地缓存。

建议目录结构：

```text
/srv/blog
  source/              # Git 仓库源码
  releases/            # 每次构建后的静态产物
    20260427-020000/
  current -> releases/20260427-020000
  logs/
    build.log
```

建议流程：

```text
cd /srv/blog/source
git fetch --all
git reset --hard origin/main
pnpm install --frozen-lockfile
pnpm build:site
copy frontend/dist -> /srv/blog/releases/{timestamp}
switch /srv/blog/current symlink atomically
reload nginx/caddy if needed
```

关键点：

- `.env` 放在服务器仓库根目录，保存 `NOTION_TOKEN`、`NOTION_DATABASE_ID`、`GITHUB_TOKEN`。
- 构建失败时不切换 `current` 软链。
- Nginx 或 Caddy 只服务 `/srv/blog/current`。
- 每次构建日志写入 `/srv/blog/logs/build-{timestamp}.log`。
- 保留最近 N 个 release，避免磁盘无限增长。
- 用 `crontab` 或 `systemd timer` 定时执行，不建议用 `pm2` 常驻跑一次性任务。

推荐定时方式：

```text
crontab
  简单，适合个人项目

systemd timer
  更稳，日志和失败状态更清晰，适合长期生产环境
```

## Deployment Option 2: GitHub Actions

核心构建一个 workflow 脚本

适用场景：

- 不想维护服务器。
- 站点可以完全静态托管。
- 希望通过 GitHub Secrets 管理 token。

现有 workflow 可以作为基础：

```text
.github/workflows/sync-static-data.yml
```

当前已具备：

- `workflow_dispatch`
- 每天 UTC 18:00 定时触发，即中国时间凌晨 2 点。
- Node 24 环境。
- pnpm install。
- Notion 数据同步。
- GitHub Projects/Profile 数据同步。
- frontend build。

下一步需要补充部署目标。建议优先级：

1. GitHub Pages：最少外部依赖，适合先跑通闭环。
2. Cloudflare Pages：CDN 和正式访问体验更好，适合正式部署。
3. Vercel/Netlify：可用，但本项目的数据同步链路由 GitHub Actions 主控会更清晰。

### GitHub Pages Flow

```text
checkout
setup node/pnpm
install
sync data
type-check
build frontend
upload frontend/dist artifact
deploy pages
```

需要配置：

- 仓库 Settings -> Pages -> Source 选择 GitHub Actions。
- workflow 设置 `permissions`：

```yaml
permissions:
  contents: read
  pages: write
  id-token: write
```

Secrets：

```text
NOTION_TOKEN
NOTION_DATABASE_ID
GH_STATS_TOKEN   # 可选，贡献图需要更完整数据时使用
```

### Cloudflare Pages Flow

```text
checkout
setup node/pnpm
install
sync data
type-check
build frontend
deploy frontend/dist to Cloudflare Pages
```

需要配置：

```text
CLOUDFLARE_API_TOKEN
CLOUDFLARE_ACCOUNT_ID
CLOUDFLARE_PROJECT_NAME
```

Cloudflare 方案的优势是正式访问体验更好，缺点是 Secrets 和项目配置比 GitHub Pages 多一点。

## Environment Strategy

本地开发：

```text
.env
```

自建服务器：

```text
/srv/blog/source/.env
```

GitHub Actions：

```text
GitHub Secrets
```

变量建议：

```env
NOTION_TOKEN=
NOTION_DATABASE_ID=
GITHUB_TOKEN=
GH_STATS_TOKEN=
BLOG_DATA_CONFIG_PATH=
```

原则：

- `GITHUB_TOKEN` / `GH_STATS_TOKEN` 只给 Node 构建脚本使用。
- 不允许出现 `VITE_GITHUB_TOKEN`、`VITE_NOTION_TOKEN`。
- generated JSON 和前端 bundle 中不得包含 token。

## Static Site Enhancements

部署阶段可以顺手补齐静态站点发布能力，但不要和部署脚本混成一团。建议后续以独立模块生成：

```text
frontend/public/robots.txt
frontend/public/sitemap.xml
frontend/public/rss.xml
```

或者构建期输出到：

```text
frontend/dist/robots.txt
frontend/dist/sitemap.xml
frontend/dist/rss.xml
```

建议优先级：

1. `robots.txt`
2. `sitemap.xml`
3. `rss.xml`
4. canonical URL / OG meta 检查
5. 部署后 URL 健康检查

## Proposed Architecture

```text
Root config / .env / GitHub Secrets
  -> pnpm sync:data
  -> data/*.json
  -> pnpm verify:frontend
  -> pnpm build:frontend
  -> frontend/dist
  -> Self-hosted current symlink
     or GitHub Pages / Cloudflare Pages
```

两个部署方案共享：

- 同一个根配置。
- 同一个数据输出目录。
- 同一个构建命令。
- 同一个 SSG 产物目录。

差异只在最后的发布目标。

## Testing and Validation

本地验证：

```text
pnpm install --frozen-lockfile
pnpm sync:data
pnpm verify:frontend
pnpm build:frontend
```

检查内容：

- `data/posts.json` 正常生成或保留。
- `data/projects.generated.json` 正常生成。
- `data/profile.generated.json` 正常生成。
- `data/github-contributions.generated.json` 正常生成。
- `frontend/dist/index.html` 存在。
- `frontend/dist/me/index.html` 包含静态 Me 页面内容。
- `frontend/dist/project/index.html` 包含静态 Project 页面内容。

GitHub Actions 验证：

- 手动触发 workflow。
- 检查 Secrets 注入是否成功。
- 检查数据同步日志。
- 检查构建产物上传或部署是否成功。
- 打开部署后的 URL。

自建服务器验证：

- 手动运行部署脚本。
- 检查 release 目录。
- 检查 `current` 软链是否切换。
- 检查 Nginx/Caddy 是否能访问最新页面。
- 模拟构建失败，确认线上版本不被破坏。

安全验证：

```text
grep token-like patterns in frontend/dist
grep token-like patterns in data
```

确认：

- 不存在 `NOTION_TOKEN` 内容。
- 不存在 `GITHUB_TOKEN` 内容。
- 不存在 `GH_STATS_TOKEN` 内容。

## Risks and Edge Cases

- GitHub Actions 默认 `github.token` 可能不能读取完整 contribution calendar，需要 `GH_STATS_TOKEN`。
- Notion API 或 GitHub API 失败可能导致数据不完整，脚本需要保留可用旧数据。
- 自建服务器构建过程中如果直接覆盖线上目录，失败时可能导致线上空白，因此必须用 release + symlink。
- GitHub Pages 有 base path 问题，如果部署到 `username.github.io/repo`，需要检查 Vite `base`。
- Cloudflare Pages 需要额外配置 API token 和项目名，Secrets 管理复杂度略高。
- SSG 动态路由依赖 `data/posts.json`，数据为空时要确认构建不会失败。
- 构建产物中的 sitemap/rss 需要正确使用生产域名，不能写死本地地址。
- 定时任务重复触发时要避免并发构建互相覆盖。
